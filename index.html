 Base de Conhecimento

      
Documentação e FAQ

      
Respostas rápidas para as dúvidas mais comuns sobre a plataforma.

    

    

      

        
        

          O Sentinel Firewall analisa padrões de tráfego em tempo real usando machine learning. Quando detecta comportamento anômalo (port scanning, brute force, DDoS), automaticamente adiciona o IP à blocklist com um TTL configurável. O sistema integra feeds de threat intelligence como abuse.ch, spamhaus e alienvault OTX para enriquecer a análise. As regras de auto-block podem ser customizadas por severidade, tipo de ataque e zona de rede.
        

      

      

        
        

          O módulo IDS/IPS opera nas camadas 3 a 7 do modelo OSI. Suporta inspeção profunda de pacotes para protocolos TCP, UDP, ICMP, HTTP/HTTPS, DNS, FTP, SSH, SMTP e TLS. As assinaturas são atualizadas diariamente via integração com Suricata ET Rules e regras customizadas podem ser escritas no formato .rules nativo.
        

      

      

        
        

          Use o Configurador de Regras nesta página para criar suas regras visualmente, depois clique em "Exportar iptables". O sistema gera comandos iptables prontos para execução. Você pode copiar e colar diretamente no terminal ou salvar como script .sh. Para persistência, use iptables-save > /etc/iptables/rules.v4 após aplicar as regras. O formato de exportação também suporta nftables e ufw.
        

      

      

        
        

          O Sentinel Firewall suporta notificações via webhook, e-mail SMTP, Slack, Telegram e PagerDuty. Configure em Configurações > Alertas no painel admin. Você pode definir regras de alerta por severidade (info, warning, critical), tipo de evento (bloqueio, scan, brute force) e frequência. Alertas críticos geram notificações imediatas; os demais são agrupados em digests configuráveis (5min, 15min, 1h).
        

      

      

        
        

          O Sentinel Firewall é projetado para alta performance. Com DPDK (Data Plane Development Kit), atinge throughput de até 40 Gbps com latência inferior a 1ms. O motor de regras usa tabelas hash otimizadas com complexidade O(1) para lookup. Em ambientes típicos (até 10k regras), o overhead é inferior a 0.1ms por pacote. O módulo IDS opera em modo passivo por padrão, sem impacto no path de dados.
        

      

      

        
        

          Sim. O Sentinel Firewall suporta deploy em cluster ativo-passivo e ativo-ativo com sincronização de estado via conntrackd. O failover automático ocorre em menos de 2 segundos. As regras são replicadas via etcd distribuído. Para ambientes Kubernetes, use o Helm chart oficial com suporte a CRDs para gerenciar regras como recursos nativos do cluster.
        

      

    

  







  

    

      
 Guia de Instalação

      
Deploy em Produção

      
Escolha o método de deploy ideal para o seu ambiente.

    

    

      

        
        
        
      

      

        
        

          

            
docker-compose.yml

            
# Sentinel Firewall - Docker Compose
version: "3.8"

services:
  sentinel-fw:
    image: sentinel/firewall:latest
    container_name: sentinel-firewall
    restart: always
    network_mode: host
    cap_add:
      - NET_ADMIN
      - NET_RAW
    volumes:
      - ./config:/app/config
      - ./data:/app/data
      - /var/log/sentinel:/app/logs
    ports:
      - "3100:3100"
    environment:
      - SENTINEL_MODE=production
      - SENTINEL_LOG_LEVEL=info
      - SENTINEL_ADMIN_PASS=SuaSenhaSegura

          

          

            
Terminal

            
# Iniciar o Sentinel Firewall
$ docker-compose up -d

# Verificar status
$ docker logs -f sentinel-firewall

# Acessar painel admin
$ curl http://localhost:3100/health

          

        

        
        

          

            
/etc/systemd/system/sentinel-fw.service

            
[Unit]
Description=Sentinel Firewall Service
After=network.target
Wants=network-online.target

[Service]
Type=simple
User=sentinel
WorkingDirectory=/opt/sentinel-firewall
ExecStart=/usr/bin/node /opt/sentinel-firewall/server.js
Restart=always
RestartSec=5
Environment=NODE_ENV=production
AmbientCapabilities=CAP_NET_ADMIN CAP_NET_RAW

[Install]
WantedBy=multi-user.target

          

          

            
Terminal

            
# Instalar e iniciar o serviço
$ sudo systemctl daemon-reload
$ sudo systemctl enable sentinel-fw
$ sudo systemctl start sentinel-fw

# Verificar status
$ sudo systemctl status sentinel-fw
$ sudo journalctl -u sentinel-fw -f

          

        

        
        

          

            
Terminal

            
# 1. Clonar repositório
$ git clone https://github.com/sentinel/firewall.git
$ cd firewall

# 2. Instalar dependências
$ npm install --production

# 3. Configurar ambiente
$ cp .env.example .env
$ nano .env

# 4. Configurar capacidades de rede
$ sudo setcap 'cap_net_admin,cap_net_raw+eip' $(which node)

# 5. Iniciar em produção
$ NODE_ENV=production node server.js

# 6. (Opcional) Usar PM2 para gerenciamento
$ pm2 start server.js --name sentinel-fw
$ pm2 save
$ pm2 startup
